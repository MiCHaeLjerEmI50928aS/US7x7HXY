How Do You Tell a Computer What to Do?

An Introduction to Visual Programming

Grade 8 Computer Science

Opening question to the class - let a few students answer. Today: why visual programming exists and why we're starting here. ~1 min

Computers can't actually think

They only do exactly what you tell them.

Nothing more. Nothing less.

Our job is to give them precise instructions.

Quick poll: who thinks computers are smart? Key point: they're obedient but literal. Can't figure anything out on their own. ~1 min

The Building Block: Transistors

A transistor is a tiny switch. It can only be in two states:

OFF
0
ON
1

A modern phone has about 15 billion transistors.

Transistor = switch. Two states only. Billions of these working together = everything your phone does. This is why computers use 0s and 1s. ~2 min

Binary: The Language of 0s and 1s

01001000 01100101 01101100 01101100 01101111

This says: "Hello"

Writing code in pure binary would be like communicating
by flicking a light switch on and off thousands of times.

This is a problem. 🤔

Each group of 8 = one letter. Build tension here: this is HARD. Set up the problem. ~2 min

The Problem

Computers speak in 0s and 1s.

Humans speak in words and ideas.

How do we bridge this gap?

Early programmers asked this same question...

Pause here. Let the question sink in. This is the core tension that drives everything we're about to learn. "Humans needed a way to talk to computers without speaking binary." ~1 min

The Solution: Abstraction

Abstraction = Hiding complexity behind something simple.

Real Life Example:

🚗

You press the gas pedal
You don't control each piston individually

Programmers did the same thing with code:

They grouped many small instructions
and gave them a name.

Relatable example: driving a car. You don't think about pistons, fuel injection, etc. You just press the pedal. The complexity is HIDDEN. Same idea in programming: hide the 0s and 1s behind simple names. ~2 min

The Tool: Functions

A Function is a named group of instructions.

sprite.x = sprite.x + 1
sprite.y = sprite.y + 0
redraw_screen()
check_boundaries()
update_position()
...(50 more lines)
→
move_steps(10)

Instead of writing 50 lines every time,
you just call the function by its name.

Functions are the building blocks of abstraction.

Key moment: connect abstraction to a concrete tool (functions). A function is like a recipe with a name. You say "make pancakes" instead of listing every step. Someone wrote those 50 lines ONCE, gave it a name, now everyone can use it. ~2 min

The Layers of Abstraction

Each layer hides the complexity of the one below it:

Visual Programming (Scratch, Blockly)
Drag blocks → they call functions for you
↑ more abstraction
Text-Based Code (Python, Java)
Write functions in human-readable words
↑ more abstraction
Machine Code (010101...)
What the computer actually runs

We're starting at the top — the most human-friendly level.

This is the staircase we built! Each step adds abstraction. Bottom = raw binary (the transistors we talked about). Middle = Python/Java uses functions with text. Top = Visual blocks that CALL those functions. We start at the top to learn the LOGIC, then can go deeper later. ~2 min

The Big Reveal: A Block is a Function

Now you understand why this works:

What You See (Block)

move (10) steps

What It Really Is (Function) 


def move_steps(sprite, steps):
    for i in range(steps):
        sprite.x += 1
        sprite.redraw()

move_steps(cat, 10)
Binary → Function → Block

When you drag a block, you're calling a function
that someone already wrote for you.

THIS IS THE PAYOFF. Everything we learned connects here. - We started with "how do we tell computers what to do?" - We saw that computers only understand binary. - We learned abstraction hides complexity. - We learned functions are named groups of code. - NOW: A Scratch block IS a function! It's abstraction in action. When they drag "move 10 steps", they're calling a function that MIT programmers wrote. ~2 min

What is Visual Programming?

"A way of creating computer programs using pictures and blocks instead of writing text code."

Now you know what's happening under the hood:

Each block is a function
that hides complex code, so you can focus on logic.

Examples:

Scratch · Blockly · MakeCode · Sprite Box

Definition from the textbook (page 52-53). But now students understand WHY blocks exist — they're abstraction! Scratch = MIT, most popular. We'll use this. ~1 min

Different systems use different ways to show logic visually:

🧭

🧱
Icon-Based

Visual Programming Application

Definition
is a software tool or interactive environment that enables users to create computer programs using graphical elements such as blocks, icons, and flowcharts.

Types of Visual Programming Applications

Visual programming is used in three main categories:

📚 EDUCATIONAL
Used in schools to introduce programming concepts in a fun and accessible way.
🎨 MULTIMEDIA
Create digital content like presentations, animations, and websites using images and sound.
🎮 VIDEO GAMES
Interactive platforms for creating games, building environments, and developing game mechanics.

The same block-based approach works for all of these!

Three categories from the curriculum: 1. Educational - learning to code in schools (like what we're doing) 2. Multimedia - creating visual content, animations, presentations 3. Video Games - building interactive experiences ~1 min

Activity: The Human Robot

The idea: I'm a robot. You're the programmers.

Robots only understand exact instructions.

Your commands:

  • "Move [number] steps"
  • "Turn left" / "Turn right"
  • "Say [message]"

Your task: Get me from here to the door using only these commands.

Physical activity, ~3-4 min. Be deliberately literal: "go to door" = "I don't understand." Let them struggle a bit, then move to next slide.

Variables: The Box

A container that stores data.

Score
10

We can change what's inside:

10
→
20

This is called Assignment (setting the value).

Think of a box with a label on it. "Score" is the name, "10" is what's inside. We can take out the 10, put in 20 - the box stays, contents change. "Assignment" = putting a value in the box. In Scratch: "set Score to 10" - that's assignment. ~2 min

See the orange box on stage? That's your variable displaying its value!

Key moments: - Show checkmark to display variable on stage - Click "set" block - variable becomes 0 - Click "change" block - variable goes up by 1 each click This is CONCRETE - they see the box, they see the number change.

Input & Output: I/O

👂
INPUT

The computer's ears

ask "What is your name?" and wait
💬
OUTPUT

The computer's mouth

say "Hello!"

Without Input, computers are useless!

INPUT = how data gets IN (keyboard, mouse, microphone). OUTPUT = how results come OUT (screen, speakers). Ears and mouth analogy - computer listens, then responds. Show how to drag "answer" into the say block's white bubble. This is the INPUT-PROCESS-OUTPUT pattern they'll use constantly!

Let's Try It: I/O in Scratch

1
Go to Sensing (light blue)
2

3
Go to Looks → Drag: say [Hello!]
4
Find answer → Put it INSIDE the say block!
5
Click the blocks! Type your name → Cat says it back! 🎉

Input (ask) → Process (store in answer) → Output (say)

HANDS-ON TIME! ~5 min Walk them through step by step. Key insight: the "answer" variable automatically stores their input. This is the INPUT-PROCESS-OUTPUT pattern!
